CobblePass - The Ultimate Battle Pass for Cobblemon

Welcome to the completely revamped CobblePass! This mod provides a deeply integrated and highly customizable battle pass system for your Cobblemon server. Engage your players with a rewarding progression system, complete with free and premium tracks, seasonal content, and a wide array of configurable rewards.

Features

  • Seasons: Create unique battle pass seasons with configurable durations and rewards.
  • Dual Reward Tracks: Implement both free and premium reward tiers.
  • Dynamic GUI: A clean, intuitive, and paginated GUI (/bp) for players to view progress and claim rewards.
  • XP Progression:
    • Gain XP from various in-game actions: catching, defeating, evolving, hatching, and trading Pokémon.
    • Two XP progression models:
      • FORMULA: XP required per level increases based on a configurable multiplier.
      • MANUAL: Manually define the exact XP required for each level.
  • Diverse Reward Types:
    • ITEM: Grant any item from Minecraft or other mods, with full NBT support.
    • POKEMON: Reward players with specific Pokémon, including level, shininess, and other attributes.
    • COMMAND: Execute any server command as a reward, with placeholders for player name and UUID.
  • Admin Management:
    • A full suite of commands for managing seasons, players, and configurations.
    • Create, start, stop, and delete seasons.
    • Modify player XP, levels, and premium status on the fly.
    • Reload configurations without a server restart.
  • Economy Integration: Charge for the premium pass using any Impactor-compatible economy plugin.
  • Highly Configurable: Tweak everything from XP values to reward templates and season settings through intuitive JSON files.
  • Data Integrity: Player data is saved reliably and includes safeguards against corruption.

Dependencies

CobblePass requires the following mods to be installed on your server:

  • Fabric API
  • Cobblemon (v1.6+)
  • Impactor (For economy features)
  • GooeyLibs

Installation

  1. Ensure you have a Fabric server running Minecraft 1.21.1 or later.
  2. Download the latest versions of CobblePass and all required dependencies.
  3. Place the .jar files into your server's mods folder.
  4. Start your server. CobblePass will generate its default configuration files in the config/cobblepass/ directory.

Getting Started: Your First Season

Setting up a new Battle Pass is easy with the in-game creation tool.

Step 1: Create the Battle Pass

  • As an operator, run the command: /battlepass create
  • This will open a GUI where you can set:
    • Duration: The length of the season in days.
    • Max Level: The highest level players can achieve.
    • Premium Toggle: Whether this season will have a premium track.
  • Click "Create Battle Pass". This generates config.json and tiers.json files and resets all player progress for a fresh start.

Step 2: Configure Your Rewards

  • Open config/cobblepass/tiers.json.
  • By default, it's filled with placeholder rewards (Apples). Customize each level's freeReward and premiumReward. See TIERS.md for detailed configuration options and examples.
  • This is where you can also define reusable reward templates!

Step 3: Configure Settings

  • Open config/cobblepass/config.json.
  • Adjust XP gain values (catchXP, defeatXP, etc.), the cost of the premium pass (premiumCost), and the XP progression model (xpProgression). See CONFIG.md for more details.

Step 4: Reload the Configuration

  • Once you've saved your changes to the JSON files, run the command: /battlepass reload
  • This will load all your new settings and rewards into the server without needing a restart.

Step 5: Start the Season!

  • When you're ready for the season to begin, run: /battlepass season start
  • The timer will start, and players can now begin earning XP!

Player Data

Player progress is stored individually in config/cobblepass/players/<uuid>.json. It is recommended not to edit these files manually.

Commands

/battlepass

  • Description: Opens the main Battle Pass interface, where you can view your progress, see available rewards, and claim them.
  • Arguments: None
  • Permission: cobblepass.command.base (defaults to none, available to all players)
  • Example:
    /battlepass

/battlepass claim <level> [is_premium]

  • Description: Allows players to claim rewards for a specific tier they have unlocked.
  • Arguments:
    • level (Required): The tier number you want to claim the reward from.
    • is_premium (Optional): A boolean (true or false) indicating whether to claim the premium reward. Defaults to false (the free reward).
  • Permission: cobblepass.claim (defaults to none, available to all players)
  • Examples:
    • To claim the free reward for tier 10:
      /battlepass claim 10
    • To claim the premium reward for tier 10 (requires premium access):
      /battlepass claim 10 true

/battlepass premium

  • Description: Provides information about the premium version of the Battle Pass and allows purchasing it if enabled.
  • Arguments:
    • info (Optional): Shows your current premium status, the cost to upgrade, and how to get it. This is the default action if no subcommand is provided.
    • buy (Optional): Purchases the premium Battle Pass if the server has configured an economy provider.
  • Permission: cobblepass.premium (defaults to op, requires administrator privileges)
  • Examples:
    • To see your premium status:
      /battlepass premium info
    • To purchase the premium pass:
      /battlepass premium buy

/battlepass addlevels [player] <levels>

  • Description: Adds a specified number of levels to a player's battle pass.
  • Arguments:
    • player (Optional): The player to add levels to. Defaults to the command executor.
    • levels (Required): The number of levels to add (integer between 1 and 100).
  • Permission: cobblepass.addlevels (defaults to op, requires administrator privileges)
  • Examples:
    • To add 5 levels to your own battle pass:
      /battlepass addlevels 5
    • To add 10 levels to a player named Steve:
      /battlepass addlevels Steve 10

/battlepass addxp <player> <amount>

  • Description: Adds a specified amount of XP to a player's battle pass.
  • Arguments:
    • player (Required): The player to add XP to.
    • amount (Required): The amount of XP to add (integer, must be at least 1).
  • Permission: cobblepass.addxp (defaults to op, requires administrator privileges)
  • Examples:
    • To add 500 XP to a player named Alex:
      /battlepass addxp Alex 500

/battlepass create

  • Description: Opens a GUI to create a new Battle Pass season. This command allows you to configure the duration, maximum level, and whether premium features are enabled. Running this command will wipe all existing player progress to ensure a clean start.
  • Arguments: None (configuration is handled via the in-game GUI).
  • Permission: cobblepass.create (defaults to op, requires administrator privileges)
  • Example:
    /battlepass create

/battlepass delete

  • Description: Deletes all Battle Pass data, including configuration files (config.json, tiers.json) and all player progress. This action is irreversible and requires a confirmation step.
  • Arguments:
    • confirm (Required for deletion): You must run the command with confirm to finalize the deletion.
  • Permission: cobblepass.delete (defaults to op, requires administrator privileges)
  • Examples:
    • To initiate the deletion process:
      /battlepass delete
    • To confirm and permanently delete the Battle Pass:
      /battlepass delete confirm

/battlepass premiumanage

  • Description: Manages premium access for players and related settings.
  • Permission: cobblepass.premiumanage (defaults to op, requires administrator privileges - level 4)

Sub-commands:

add <player>

  • Description: Grants premium access to a specific player.
  • Arguments:
    • player (Required): The player to grant premium access to.
  • Example:
    /battlepass premiumanage add Steve

remove <player>

  • Description: Revokes premium access from a specific player.
  • Arguments:
    • player (Required): The player to revoke premium access from.
  • Example:
    /battlepass premiumanage remove Steve

bulk add <players>

  • Description: Grants premium access to multiple players at once.
  • Arguments:
    • players (Required): The target players (can use selectors like @a).
  • Example:
    /battlepass premiumanage bulk add @a[tag=vip]

bulk remove <players>

  • Description: Revokes premium access from multiple players at once.
  • Arguments:
    • players (Required): The target players (can use selectors like @a).
  • Example:
    /battlepass premiumanage bulk remove @a[tag=expired_vip]

mode set <mode>

  • Description: Sets the server's premium mode.
  • Arguments:
    • mode (Required): The premium mode to set. Valid options are ECONOMY, PERMISSION, or DISABLED.
  • Example:
    /battlepass premiumanage mode set ECONOMY

mode get

  • Description: Displays the current premium mode and its description.
  • Arguments: None
  • Example:
    /battlepass premiumanage mode get

status <player>

  • Description: Checks the premium status of a specific player.
  • Arguments:
    • player (Required): The player whose status you want to check.
  • Example:
    /battlepass premiumanage status Alex

migrate

  • Description: Initiates a migration process for all players' premium statuses. This is useful to ensure consistency after configuration changes.
  • Arguments: None
  • Example:
    /battlepass premiumanage migrate

/battlepass reload

  • Description: Reloads all Battle Pass configuration files, including config.json, tiers.json, and lang.json. This is useful for applying changes without restarting the server.
  • Arguments: None
  • Permission: cobblepass.reload (defaults to op, requires administrator privileges - level 4)
  • Example:
    /battlepass reload

/battlepass season

  • Description: Manages Battle Pass seasons, allowing you to start, stop, or transition between them.
  • Permission: cobblepass.season (defaults to op, requires administrator privileges - level 4)

Sub-commands:

start

  • Description: Starts a new Battle Pass season if one is not already active.
  • Arguments: None
  • Example:
    /battlepass season start

stop

  • Description: Immediately stops the current Battle Pass season. Player progress is paused but not reset.
  • Arguments: None
  • Example:
    /battlepass season stop

endseason

  • Description: Initiates the process of ending the current season and starting a new one. This command opens a GUI to configure the reset process, including options for preserving premium status, creating backups, and validating data. This is the recommended and safest way to transition between seasons.
  • Arguments: None
  • Example:
    /battlepass season endseason

Configuration

This file documents the main configuration settings for the CobblePass mod, found in config/cobblepass/config.json.

Main Settings

maxLevel

  • Description: The maximum level a player can achieve in the battle pass.
  • Default Value: 100
  • Example:
    "maxLevel": 100

catchXP

  • Description: The amount of XP awarded for catching a Pokémon.
  • Default Value: 100
  • Example:
    "catchXP": 100

defeatXP

  • Description: The amount of XP awarded for defeating a Pokémon.
  • Default Value: 50
  • Example:
    "defeatXP": 50

evolveXP

  • Description: The amount of XP awarded for evolving a Pokémon. This value is not in the default config.json but can be added.
  • Default Value: 150
  • Example:
    "evolveXP": 150

hatchXP

  • Description: The amount of XP awarded for hatching a Pokémon from an egg. This value is not in the default config.json but can be added.
  • Default Value: 200
  • Example:
    "hatchXP": 200

tradeXP

  • Description: The amount of XP awarded for trading a Pokémon. This value is not in the default config.json but can be added.
  • Default Value: 250
  • Example:
    "tradeXP": 250

seasonDurationDays

  • Description: The duration of a battle pass season in days.
  • Default Value: 60
  • Example:
    "seasonDurationDays": 60

currentSeason

  • Description: The identifier for the current season. This is managed internally and should not be manually edited unless you know what you are doing.
  • Default Value: 0
  • Example:
    "currentSeason": 0

seasonStartTime

  • Description: The Unix timestamp (in milliseconds) when the current season started. This is managed internally. A value of 0 indicates no season is active.
  • Default Value: 0
  • Example:
    "seasonStartTime": 1672531200000

seasonEndTime

  • Description: The Unix timestamp (in milliseconds) when the current season will end. This is managed internally. A value of 0 indicates no season is active.
  • Default Value: 0
  • Example:
    "seasonEndTime": 1677715200000

enablePermissionNodes

  • Description: If true, enables permission nodes for various commands and features. If false, permissions are not checked.
  • Default Value: true
  • Example:
    "enablePermissionNodes": true

XP Progression (xpProgression)

The xpProgression object controls how the XP required for each level is calculated. You can use a formula-based approach or define manual XP values for each level.

  • Example xpProgression object:
    "xpProgression": {
  "mode": "FORMULA",
  "xpPerLevel": 1000,
  "xpMultiplier": 1.1,
  "manualXpValues": {
    "1": 500,
    "2": 1000,
    "3": 1500
  }
}

mode

  • Description: Determines the method for calculating XP requirements.
    • FORMULA: Uses a formula where the XP for a level is xpPerLevel * (level * xpMultiplier).
    • MANUAL: Uses the specific values defined in manualXpValues. If a level is not defined in the map, leveling up will be effectively disabled for that level.
  • Default Value: "FORMULA"
  • Example:
    "mode": "MANUAL"

xpPerLevel

  • Description: The base XP required per level when using the FORMULA mode.
  • Default Value: 1000
  • Example:
    "xpPerLevel": 1000

xpMultiplier

  • Description: The multiplier applied to the level when calculating XP requirements in FORMULA mode. This creates an escalating XP curve.
  • Default Value: 1.1
  • Example:
    "xpMultiplier": 1.2

manualXpValues

  • Description: A map where you can define the exact amount of XP required for each specific level. This is only used when mode is set to MANUAL. The key is the level (as a string), and the value is the XP required.
  • Default Value: {} (empty object)
  • Example:
    "manualXpValues": {
  "1": 500,
  "2": 1000,
  "3": 1500,
  "4": 2200,
  "5": 3000
}

Premium Pass (premiumConfig)

The premiumConfig object allows you to configure how players obtain premium access to the battle pass. This object is optional; if it's not present in your config.json, default values will be used.

  • Example premiumConfig object:
    "premiumConfig": {
  "mode": "economy",
  "permissionNode": "cobblepass.premium",
  "economyEnabled": true,
  "premiumCost": 1000,
  "autoRenew": false,
  "preserveOnSeasonChange": true
}

mode

  • Description: Determines the method for granting premium access.
    • economy: Players can purchase premium access using an economy plugin (e.g., Vault). The cost is defined by premiumCost.
    • permission: Premium access is granted to players who have a specific permission node, defined by permissionNode.
    • disabled: Premium access is disabled. All players are considered premium, and no purchase or permission is required.
  • Default Value: "economy"
  • Example:
    "mode": "permission"

permissionNode

  • Description: The permission node required for premium access when mode is set to permission.
  • Default Value: "cobblepass.premium"
  • Example:
    "permissionNode": "my.custom.permission"

economyEnabled

  • Description: A legacy setting that is largely controlled by the mode. When mode is economy, this is effectively true.
  • Default Value: true
  • Example:
    "economyEnabled": true

premiumCost

  • Description: The amount of in-game currency required to purchase premium access when mode is set to economy.
  • Default Value: 1000
  • Example:
    "premiumCost": 2500

autoRenew

  • Description: If set to true, the server will attempt to automatically charge players and renew their premium status at the beginning of a new season. This only applies if mode is economy.
  • Default Value: false
  • Example:
    "autoRenew": true

preserveOnSeasonChange

  • Description: If set to true, players who had premium access in the previous season will automatically be granted premium access in the new season at no cost. If autoRenew is also true, the renewal charge will still be attempted.
  • Default Value: true
  • Example:
    "preserveOnSeasonChange": false

Season Reset (seasonResetConfig)

The seasonResetConfig object allows you to configure how season resets are handled. This includes settings for automatic backups, preservation of premium status, and other safety features. This object is optional; if it's not present in your config.json, default values will be used.

  • Example seasonResetConfig object:
    "seasonResetConfig": {
  "autoBackupOnReset": true,
  "defaultPreservationMode": "PRESERVE_ALL",
  "broadcastTransitionMessages": true,
  "backupRetentionDays": 30,
  "requireConfirmation": true,
  "validateBeforeReset": true
}

autoBackupOnReset

  • Description: If true, the server will automatically create a backup of player data before a season reset is performed. This is a critical safety feature to prevent data loss.
  • Default Value: true
  • Example:
    "autoBackupOnReset": true

defaultPreservationMode

  • Description: Determines how premium status is handled for players during a season reset.
    • PRESERVE_ALL: All players with premium status will keep it in the new season.
    • PRESERVE_PAID: Only players who paid for premium (if using an economy provider) will keep it.
    • NONE: No premium status is preserved; all players will need to re-acquire it.
  • Default Value: "PRESERVE_ALL"
  • Example:
    "defaultPreservationMode": "PRESERVE_PAID"

broadcastTransitionMessages

  • Description: If true, global messages will be broadcast to all players announcing the end of a season and the start of a new one.
  • Default Value: true
  • Example:
    "broadcastTransitionMessages": true

backupRetentionDays

  • Description: The number of days to keep automatic backups before they are deleted. This helps manage disk space while retaining recent backups. The value must be between 1 and 365.
  • Default Value: 30
  • Example:
    "backupRetentionDays": 15

requireConfirmation

  • Description: If true, any command that initiates a season reset will require a second confirmation command to proceed. This prevents accidental resets.
  • Default Value: true
  • Example:
    "requireConfirmation": true

validateBeforeReset

  • Description: If true, the system will perform a series of validation checks before executing a season reset to ensure that the configuration is valid and that the reset can be performed safely.
  • Default Value: true
  • Example:
    "validateBeforeReset": true

GUI Customization

The gui.json file allows for complete customization of the Battle Pass GUI. You can change the title, layout, and items displayed.

Main Properties

title

  • Type: String
  • Description: The title of the GUI. This can be a plain string or a language key from lang.json.
  • Example: "title": "My Awesome Battle Pass" or "title": "lang.gui.title"

enableCustomGui

  • Type: Boolean
  • Description: Enables or disables the custom GUI. If set to false, a default, hard-coded GUI will be used.
  • Example: "enableCustomGui": true

structure Object

The structure object defines the layout and the items within the GUI. It consists of two main parts: structure and ingredients.

structure Array

  • Type: Array of Strings
  • Description: This array defines the visual layout of the GUI. Each string in the array represents a row, and each character represents a slot. The characters are mapped to items in the ingredients object. A space character (' ') will be treated as an empty slot.
  • Example:
    "structure": [
  "#########",
  "#ifbpr L#",
  "#       #",
  "#<     >#"
]

ingredients Object

  • Type: Object
  • Description: This object maps the characters used in the structure array to specific item definitions. Each key is a character, and the value is an object defining the item's properties.
  • Example:
    "ingredients": {
  "#": {
    "type": "STATIC_ITEM",
    "material": "minecraft:light_gray_stained_glass_pane",
    "name": "lang.gui.decoration.border",
    "hideTooltip": true
  },
  "f": {
    "type": "FREE_REWARD_PLACEHOLDER"
  },
  "r": {
    "type": "PREMIUM_REWARD_PLACEHOLDER"
  },
  "<": {
    "type": "NAVIGATION_PREVIOUS",
    "material": "minecraft:arrow",
    "name": "lang.gui.navigation.previous"
  }
}

In the example above:

  • The # character creates a border.
  • i, f, b, p, r, and L are placeholders for various dynamic or static items.
  • The space characters create an empty row in the middle.
  • < and > are navigation buttons.

GuiIngredient Types

This section details the different types of ingredients you can use in the ingredients object of your gui.json.

STATIC_ITEM

  • Purpose: A simple, non-interactive item used for decoration, such as borders or fillers.

  • Configurable Properties:

    • material (String, Optional, Default: "minecraft:gray_stained_glass_pane"): The Minecraft item ID to display.
    • name (String, Optional): The display name of the item. Supports color codes and language keys.
    • lore (List of Strings, Optional): A list of strings to be used as the item's lore.
    • customModelData (Integer, Optional): The custom model data for the item, for use with resource packs.
    • hideTooltip (Boolean, Optional, Default: true): If true, hides the default item tooltip (e.g., "minecraft:stone").

  • Example:

    "b": {
  "type": "STATIC_ITEM",
  "material": "minecraft:black_stained_glass_pane",
  "name": " "
}

  • Purpose: A placeholder that will be dynamically replaced by a free reward item from tiers.json.

  • Purpose: A placeholder that will be dynamically replaced by a premium reward item from tiers.json.

  • Example:

  "material": "minecraft:experience_bottle",

  • Purpose: Displays the player's overall Battle Pass progression.

FREE_REWARDS_LABEL

CUSTOM_ITEM

  • Purpose: A fully customizable item that can be used for decoration or to execute a command when clicked.

  • Configurable Properties:

    • material (String, Required): The Minecraft item ID to display.
    • name (String, Optional): The display name.
    • lore (List of Strings, Optional): The item's lore.
    • customModelData (Integer, Optional): The custom model data.
    • command (String, Optional): The command to execute on click. Placeholders like {player} are supported.
    • permission (String, Optional): A permission node required to execute the command.

Lang File (lang.json)

The lang.json file is used to customize all the text displayed by the CobblePass mod. This allows server owners to translate messages into different languages or change the wording to better fit their server's theme.

Placeholders

The lang file uses placeholders to insert dynamic values into messages, such as player names, numbers, or time. There are two types of placeholders: indexed and named.

Indexed Placeholders

Indexed placeholders are replaced by values in the order they are provided in the code. They are useful for simple substitutions.

  • %s - Replaced by a string (text).
  • %d - Replaced by an integer (number).

Example:

"lang.command.xp_gained": "§b[BattlePass] §fYou gained §e%d§f XP! (§e%d§f/§e%d§f)"

In this example, the three %d placeholders would be replaced by the amount of XP gained, the current total XP, and the XP needed for the next level, respectively.

Named Placeholders

Named placeholders are replaced by values based on their name. This can make the lang file easier to read and manage, especially for complex messages.

  • {placeholder} - Replaced by the value associated with "placeholder".

Example:

While the current lang.json doesn't use named placeholders, a future implementation might look like this:

"lang.welcome_message": "Welcome, {player_name}! You are on level {level}."

Here, {player_name} would be replaced by the player's name and {level} by their current Battle Pass level.

Tiers Configuration (tiers.json)

The tiers.json file defines the rewards for each level of your Battle Pass. It consists of a main JSON object containing a tiers array. Each element in the array represents a single tier.

Basic Structure

The root of the file is a JSON object that can contain two main properties: templates and tiers.

{
  "templates": {
    "template_name": { ... }
  },
  "tiers": [
    { ... tier object ... },
    { ... tier object ... }
  ]
}

Tier Object

Each tier object in the tiers array has the following structure:

  • level (number, required): The level this tier represents.
  • freeReward (object, optional): The reward given to all players who reach this level.
  • premiumReward (object, optional): The reward given only to players with a premium pass who reach this level.

Example:

{
  "level": 1,
  "freeReward": {
    "type": "ITEM",
    "data": {
      "id": "cobblemon:poke_ball",
      "Count": 5
    }
  },
  "premiumReward": {
    "type": "POKEMON",
    "data": {
      "species": "charmander",
      "level": 5,
      "shiny": true
    }
  }
}

Templates

The templates feature allows you to define reusable reward structures, which can simplify your tiers.json file and reduce duplication. This is especially useful for rewards that appear in multiple tiers.

Defining Templates

Templates are defined in the templates object at the root of your tiers.json file. Each key in this object is a template name, and its value is a standard reward object.

Example:

{
  "templates": {
    "common_item_bundle": {
      "type": "ITEM",
      "data": {
        "id": "cobblemon:poke_ball",
        "Count": 10
      }
    },
    "rare_pokemon": {
      "type": "POKEMON",
      "data": {
        "species": "dratini",
        "level": 20
      }
    }
  },
  "tiers": [
    ...
  ]
}

Using Templates in Tiers

To use a template in a tier, create a reward object with a single property: $ref. The value of $ref should be a string that points to the template you want to use, in the format #/templates/template_name.

Example:

{
  "level": 1,
  "freeReward": {
    "$ref": "#/templates/common_item_bundle"
  }
},
{
  "level": 10,
  "premiumReward": {
    "$ref": "#/templates/rare_pokemon"
  }
}

When the configuration is loaded, the $ref will be replaced by the content of the corresponding template.

Reward Types

Each reward object (freeReward or premiumReward) must have a type and a data object.

ITEM

The ITEM reward type is used to give players any item from Minecraft, Cobblemon, or any other mod. The system handles both minecraft: and cobblemon: item IDs, as well as items from other mods.

Properties:

  • id (string, required): The item's identifier (e.g., minecraft:diamond, cobblemon:poke_ball).
  • Count (number, optional): The number of items to give. Defaults to 1.
  • Any other valid NBT data for the item.

Example:

"freeReward": {
  "type": "ITEM",
  "data": {
    "id": "minecraft:iron_ingot",
    "Count": 10
  }
}

POKEMON

The POKEMON reward type gives the player a Pokémon with specified attributes.

Properties:

  • species (string, required): The species of the Pokémon (e.g., pikachu, eevee).
  • level (number, optional): The level of the Pokémon.
  • shiny (boolean, optional): Whether the Pokémon should be shiny. Defaults to false.
  • ability (string, optional): A specific ability for the Pokémon.

Example:

"premiumReward": {
  "type": "POKEMON",
  "data": {
    "species": "eevee",
    "level": 15,
    "shiny": true
  }
}

COMMAND

The COMMAND reward type executes a server command. This is useful for integrating with other plugins, giving permissions, or running custom functions.

Properties:

  • command (string, required): The command to execute. You can use placeholders:
    • %player%: The player's username.
    • %uuid%: The player's UUID.
  • data (object, required): An object to define how the reward is displayed in the GUI.
    • id (string, required): The item ID to display as an icon for the reward.
    • display_name (string, optional): A custom name to show for the reward in the GUI.

Example:

"premiumReward": {
  "type": "COMMAND",
  "command": "msg %player% You've unlocked a special command reward!",
  "data": {
    "id": "minecraft:paper",
    "display_name": "Special Message"
  }
}
# Enhanced Season Command Usage Guide

The enhanced SeasonCommand provides comprehensive season management capabilities with confirmation dialogs, progress feedback, and detailed reporting.

## Commands

### Basic Season Management

#### Start a Season

/battlepass season start

- Starts a new battle pass season
- Requires operator permission (level 4)
- Only works if no season is currently active

#### Stop a Season

/battlepass season stop

- Stops the current battle pass season
- Requires operator permission (level 4)
- Only works if a season is currently active

### Enhanced Season Reset

#### End Season with Reset

/battlepass season endseason

- Opens an interactive GUI for season reset configuration
- Requires operator permission (level 4)
- Must be run by a player (not console)
- Only works if a season is currently active

## Season Reset GUI

When you run `/battlepass season endseason`, you'll see a confirmation GUI with the following options:

### Premium Preservation Modes

1. **Preserve All Premium** (Diamond)
   - Keeps premium status for all current premium players
   - Default and recommended option

2. **Sync from Permissions** (Redstone)
   - Grants premium based on permission nodes only
   - Useful for permission-based premium systems

3. **Preserve + Sync** (Emerald)
   - Keeps existing premium AND syncs from permissions
   - Best of both worlds approach

4. **No Preservation** (Coal)
   - All players lose premium status on reset
   - Use with caution

### Additional Options

- **Broadcast Messages**: Send season transition messages to all players
- **Create Backup**: Create timestamped backups before reset (recommended)
- **Validate Before Reset**: Run pre-operation validation checks (recommended)

### Confirmation

- **Green Confirm Button**: Proceed with the season reset
- **Red Cancel Button**: Cancel the operation

## Progress Feedback

During a season reset, you'll receive real-time progress updates:

1. **Initiating**: Operation startup
2. **Creating backup**: Player data backup creation (20%)
3. **Preserving premium status**: Premium player list storage (40%)
4. **Clearing player progress**: Actual data reset (60%)
5. **Restoring premium status**: Premium restoration based on selected mode (80%)
6. **Finalizing**: Operation completion (100%)

## Completion Summary

After a successful reset, you'll receive a detailed summary including:

- Total players reset
- Premium players preserved/restored
- Number of backup files created
- Operation duration
- New season information

## Error Handling

The system includes comprehensive error handling for:

- **No Active Season**: Cannot reset if no season is running
- **Operation in Progress**: Prevents concurrent reset operations
- **Backup Failures**: Cancels reset if backup creation fails
- **Validation Errors**: Pre-operation checks prevent unsafe operations
- **Rollback Support**: Automatic rollback on critical failures

## Language Customization

All messages can be customized in your `lang.json` file. Key message categories:

- `lang.season.reset.confirm.*` - Confirmation dialog messages
- `lang.season.reset.progress.*` - Progress update messages
- `lang.season.reset.complete.*` - Completion summary messages
- `lang.season.reset.error.*` - Error messages
- `lang.season.premium.*` - Premium preservation messages

## Best Practices

1. **Always Create Backups**: Keep the backup option enabled
2. **Use Validation**: Keep pre-reset validation enabled
3. **Choose Appropriate Preservation Mode**: Consider your server's premium system
4. **Inform Players**: Use broadcast messages to keep players informed
5. **Monitor Progress**: Watch the progress messages for any issues
6. **Test First**: Try the process on a test server before production use

## Troubleshooting

### Common Issues

1. **"No active season to reset"**
   - Start a season first with `/battlepass season start`

2. **"Season transition already in progress"**
   - Wait for the current operation to complete
   - Check server logs for any stuck operations

3. **"Failed to create player data backup"**
   - Check file system permissions
   - Ensure sufficient disk space
   - Verify the config directory is writable

4. **GUI doesn't open**
   - Command must be run by a player, not console
   - Ensure you have operator permissions
   - Check that GooeyLibs is properly installed

### Recovery

If a reset fails:
1. Check the error message for specific details
2. Look for backup files in the config directory
3. Use the rollback information if provided
4. Contact support with server logs if needed

## Technical Details

The enhanced SeasonCommand integrates with:
- **SeasonManager**: Central coordination of season operations
- **SeasonResetHandler**: Actual reset operation execution
- **PremiumPreservationService**: Premium status management
- **SeasonTransitionMessenger**: Player communication
- **LangManager**: Localized messaging system

All operations are asynchronous and include proper error handling and rollback capabilities.

## Simple Battle Pass

This example demonstrates a basic Battle Pass with 5 levels, linear progression (the same XP is required for each level), and only free rewards. It's a great starting point for servers wanting a simple, no-fuss engagement system.

### How It Works Together

1.  **`config.json`**: This file is the brain of your Battle Pass. It sets the `maxLevel` to 5 and defines that each level requires `100` XP. It also specifies how players earn that XP (e.g., `10` XP for catching a Pokémon).
2.  **`tiers.json`**: This file is the heart of your rewards system. It lists the specific items players will receive for reaching each of the 5 levels. Since this is a simple setup, we only use `freeReward`.
3.  **`gui.json`**: This file controls the visual presentation. It creates a clean, single-row interface where the five reward items from `tiers.json` are displayed, making it easy for players to track their progress and see what's next.

This combination results in a straightforward and engaging Battle Pass where players level up by playing and are rewarded with items at each milestone.

Advanced Battle Pass

This example showcases a sophisticated Battle Pass designed for servers that want fine-grained control and a premium experience. It features manual XP progression, permission-based premium access, command-based rewards, reward templates for efficiency, and a highly customized GUI.

How It Works Together

  1. config.json: This configuration sets the xpProgression mode to MANUAL, meaning administrators must grant XP or levels to players via commands (e.g., /pass addxp or /pass addlevels). The premiumConfig is set to PERMISSION mode, requiring players to have the cobblepass.premium permission node to access premium rewards. This is ideal for servers that integrate with donation stores or other permission-granting plugins.
  2. tiers.json: This file defines a rich reward structure with both free and premium tracks. It uses a template for the "Rare Candy" reward to avoid duplicating the item definition. It also includes a COMMAND reward, which runs a server command (kit give legendary {player}) to grant the player a special kit when they claim the final premium reward.
  3. gui.json: This file creates a visually distinct, two-row GUI to separate free and premium rewards. It uses a CUSTOM_ITEM as a decorative centerpiece and a COMMAND_BUTTON that allows players with the cobblepass.buy.premium permission to run the /pass premium command directly from the GUI, providing a seamless upgrade path.

This combination creates a flexible and powerful Battle Pass system that can be tailored to specific server needs, offering a clear distinction between free and premium content while giving administrators full control over player progression.
